## Activity & Service & BroadcastReceiver & ContentProvider
각 컴포넌트는 하나의 독립적인 형태로, 고유의 기능을 수행하며 Intent 를 통해 서로 상호 작용한다.  

## 안드로이드 애플리케이션 구조
안드로이드 애플리케이션을 우리에게 친숙한 MVC 모델로 구성된다(Model, View, Control). 최근들어 MVVM이 인기지만 MVC가 기본이라 볼 수 있다.
+ VIEW  
  화면 구성을 담당하는 영역으로서, 이 영역은 View클래스를 상속하는 클래스들을 사용해서 구성할 수 있다.
+ CONTROL  
  VIEW와 MODEL을 서로 연결하고 제어하는 영역으로서, 이 영역에 엑티비티(프래그먼트), 서비스, 브로드케스트 리시버가 포함되어있다.
+ MODEL  
  애플리케이션의 데이터를 저장하는 역할을 담당하는 영역으로서, SQLite/Room/File i/o 등 콘텐츠 프로바이더가 이 영역에 해당한다.

### Activity
사용자와 직접적인 상호작용하는 담당하는 인터페이스로 사용자에게 보여지는 단일화면이다. 하나의 애플리케이션에는 반드시 하나이상의 엑티비티가 있어야 하며,  
intent를 통해 다른 애플리케이션의 엑티비티를 호출하는것이 가능하다. 또한 한화면에 2개에상의 엑티비티를 띄울수 없기 때문에 프래그먼트를 사용해 화면을 분할시키기도 한다.

### Service
사용자의 인터페이스(UI)를 방해하지 않고 눈에 보이지 않는 곳(백그라운드)에서 작업을 처리하는 컴포넌트이다.  모든 서비스는 Sevice클래스를 상속받아서 작성한다.  
대표적으로 음악재생(애플리케이션이 종료되어도 이미 시작한 서비스는 백그라운드에서 계속 동작)이 있다.  서비스는 메인 스레드에서 동작하기 때문에 서비스 내에서 별도의 스레드를 생성해 작업을 처리해야한다.(무언가 존재하는 별도의 스레드에서 동작하는것이 아님!)  
네트워크와 연동이 가능하고, 별도의 UI없이 백그라운드에서 수행된다. 다만 엑티비티와 서비스는 UI스레드 라고 불리는 동일한 애플리케이션 스레드로 실행된다.

### BroadcastReceiver
안드로이드 OS로부터 발생하는 각종 이벤트와 정보를 받아와 핸들링 하는 컴포넌트 이다. 대표적으로 문자메시지가 있으며,  
이외에도 안드로이드 디바이스의 시스템 부팅시 앱 초기화, 네트워크 끊김, 베터리 부족 알림 등 특수한 이벤트에대한 처리를 담당한다.  (ex. 안드로이드 OS에 문자 메시지가 오면, 메시지가 도착했다는 정보를 OS내에 모든 앱에 BroadCast(방송) 하고, 이 메시지를 받기위해 앱에서 Broadcast Receiver를 구현하면 이를 통해 특정 이벤트를 처리할 수 있다.)  
거의 대부분 UI를 가지지 않고, 특정상황을 제외하고 브로드 케스트는 시스템에서 시작한다.  
정리하자면 단말기에서 발생하는 일 중에서 애플리케이션이 알야하 하는 상황이 발생하면 방송을 해주고, 수신기를 통해 상황을 감지하고 적절한 작업을 수행하도록 한다.

### ContentProvider
데이터를 관리하고 다른 애플리케이션의 데이터를 제공하는 데 사용되는 컴포넌트 이다. 특정 애플리케이션이 다루고 있는 DB를 공유하기위해 사용하고, 애플리케이션간의 데이터 공유를 위해 표준화된 인터페이스를 사용한다.  SQLite, Room, web, File i/o 등을 통해 데이터를 관리하며 작은 데이터들은 intent로 데이터를 공유하지만 ContentProvider는 음악파일,사진파일 등과 같이 용량이 큰 데이터들을 공유하는데 적합하다.  Provider는 데이터의 Read(읽기), Write(쓰기)에 대한 권한이 있어야 접근이 가능하고, CRUD원칙을 준수한다.
